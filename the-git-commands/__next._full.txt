1b:T3f74,<h2>Git</h2>
<p><strong>Git</strong> is a distributed version control system that allows developers to track changes in source code during software development.
Created by <em>Linus Torvalds</em> in 2005, it helps manage project history, collaborate with others, and roll back to previous versions if needed.
Git stores data as snapshots of the entire project, unlike traditional systems that track file-by-file changes.</p>
<h3>Key feature of Git</h3>
<ul>
<li><strong>Branching</strong>: Developers can create branches to experiment with changes without affecting the main codebase.</li>
<li><strong>Merging</strong>: Combines changes from different branches or developers.</li>
<li><strong>Commit history</strong>: Records snapshots of changes, enabling easy tracking of project progress.</li>
<li><strong>Collaboration</strong>: Supports teamwork by allowing multiple contributors to work on the same project from different locations.</li>
</ul>
<h3>Let's setup Git</h3>
<p><strong>Windows</strong>: <strong><a href="https://git-scm.com/downloads/win">Download Git</a></strong> and run the installer. You can use the default settings.
<strong>MacOS</strong>: Git is usually pre-installed. If not, install it using Homebrew</p>
<pre><code class="hljs language-bash">brew install git
</code></pre>

<pre><code class="hljs language-bash"><span class="hljs-built_in">sudo</span> apt update

</code></pre>
<p>Now, You can list all branches using <code>git branch</code></p>
<p>You have learnt that how to create branch.
What is the usecase of it? 🧐</p>
<p><strong>Branch</strong>, is like a separate copy of your local codebase, in the same local setup.</p>
<p>When there is only one branch that is <code>main</code>, your all <code>commit</code>s will be pushed to <code>main</code>.
<img src="/internals/git/only-main.png" alt="only main"></p>
<p><strong>Let's discuss the usecase</strong> 🤓
Working in a team means multiple local setup. Which mean multiple collaborator to push code.
But there is a problem with many collaborator.</p>
<p><img src="/internals/git/conflict-1.png" alt="conflict with single branch">
Here is the breakdown, <code>User_1</code> has pushed his commit before <code>User_2</code> on the <code>main</code> branch.
But the <code>User_2</code> still in initial commit state. His local setup is not aware of changes which is done by <code>User_1</code>.</p>
<p><code>User_2</code>s local setup has no idea what was the change <code>User_1</code> made may be it is a adding code, creating file, modifying a file or deleting a file.</p>
<p>Suppose, <code>User_2</code> has modified a file which is deleted or modified in second commit of <code>User_1</code>.
And already pushed to main.</p>
<p>The file is changed in <code>main</code> branch. If the <code>User_2</code> pushes the change. It will show a conflict.</p>
<p>Because of modifying same file at the same line
<img src="/internals/git/confusion.png" alt="confusion"></p>
<p>Now somehow <code>User_2</code> pushes his change to <code>main</code> next time <code>User_1</code> will have to face conflicts.</p>
<p><strong>How branching works?</strong>
Before creating a branch make sure you are in <code>main</code> branch. 😀</p>
<pre><code class="hljs language-bash"><span class="hljs-comment"># create branch</span>
git branch new-feature

<span class="hljs-comment"># navigate to branch</span>
git checkout new-feature
</code></pre>
<p>Alternate</p>
<pre><code class="hljs language-bash">git checkout -b new-feature
</code></pre>
<blockquote>
<p><code>-b</code> is a flag to create branch</p>
</blockquote>
<p>Check the current branch <code>git branch</code></p>
<p>Now you are able to make changes on newly created branch ( eg. <code>new-feature</code> ).
Once the changes have done, you can push the changes to remote.</p>
<pre><code class="hljs language-bash">git add .

git commit -m<span class="hljs-string">"change"</span>

git push -u origin new-feature
</code></pre>
<div class="alert alert-warning"><div class="alert-header"><span class="alert-icon"><svg xmlns="http://www.w3.org/2000/svg" width="18" height="18" viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2" stroke-linecap="round" stroke-linejoin="round" aria-hidden="true" class="alert-icon-svg"><path d="m21.73 18-8-14a2 2 0 0 0-3.46 0l-8 14A2 2 0 0 0 4 21h16a2 2 0 0 0 1.73-3"></path><path d="M12 9v4"></path><path d="M12 17h.01"></path></svg></span><span class="alert-label">Warning</span></div>
<p>Default upstream is set to <code>main</code></p>
<p><code>-u</code> flag is used change the upstream so that next time when you push the code from <code>new-feature</code> branch<br>
You can use the <code>git push</code> command.</p>
<p><code>origin</code> creates branch on remote for current push, if not existed.</p>
<p>Alternate</p>
<pre><code class="hljs language-bash">git push origin main
</code></pre>
<p>Will push changes on remote but does not change upstream.<br>
In this case if you switch back to main branch for few changes to push directly into main.\</p>
<pre><code class="hljs language-bash"><span class="hljs-comment"># switch branch</span>
git checkout main

<span class="hljs-comment"># push changes from main</span>
git push
</code></pre>

<p><strong>Work is not over yet</strong> 😉
You have just pushed changes to new branch ( eg. <code>new-feature</code> ), It does not mean all the changes will reflect to <code>main</code> branch.</p>
<p>What!
— "<em>Is there any bug?</em>" 🥲</p>
<p>Because, Your <code>new-feature</code> branch is independent.
You will have to say the remote — my branch is done with changes, you please merge it.</p>
<p>Kidding, Just open up your repo in GitHub and raise a <code>pull request</code>. Once it merged to <code>main</code> branch your changes will reflect on <code>main</code>.</p>
